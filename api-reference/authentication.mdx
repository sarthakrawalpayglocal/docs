Every request to the PayGlocal API must include a valid API key. Authentication is handled via the HTTP `Authorization` header using the `Bearer` scheme. You receive separate API keys for the sandbox and production environments — keep them stored securely and never expose them in client-side code or public repositories.

## Obtaining your API keys

API keys are generated from the PayGlocal merchant dashboard. Navigate to **Settings → API Keys** to create or rotate your keys. Each environment (sandbox and production) has its own key; a sandbox key will be rejected on the production endpoint and vice versa.

<Warning>
Treat your API keys like passwords. Do not commit them to source control, log them, or expose them in frontend JavaScript. Use environment variables or a secrets manager to inject them at runtime.
</Warning>

## Setting the Authorization header

Include your API key on every request using the `Authorization` header with the `Bearer` prefix:

```
Authorization: Bearer {API_KEY}
```

Replace `{API_KEY}` with your actual key string. You must also include `Content-Type: application/json` on all requests that send a body.

<CodeGroup>

```bash curl
curl --request POST \
  --url https://api.sandbox.payglocal.in/gl/v1/payments/initiate \
  --header "Authorization: Bearer YOUR_SANDBOX_API_KEY" \
  --header "Content-Type: application/json" \
  --data '{
    "merchantId": "MERCH_001",
    "merchantTxnId": "ORD_20240115_001",
    "merchantUniqueId": "UNIQUE_20240115_001"
  }'
```

```javascript Node.js
const response = await fetch(
  "https://api.sandbox.payglocal.in/gl/v1/payments/initiate",
  {
    method: "POST",
    headers: {
      Authorization: "Bearer YOUR_SANDBOX_API_KEY",
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      merchantId: "MERCH_001",
      merchantTxnId: "ORD_20240115_001",
      merchantUniqueId: "UNIQUE_20240115_001",
    }),
  }
);
```

```python Python
import requests

response = requests.post(
    "https://api.sandbox.payglocal.in/gl/v1/payments/initiate",
    headers={
        "Authorization": "Bearer YOUR_SANDBOX_API_KEY",
        "Content-Type": "application/json",
    },
    json={
        "merchantId": "MERCH_001",
        "merchantTxnId": "ORD_20240115_001",
        "merchantUniqueId": "UNIQUE_20240115_001",
    },
)
```

</CodeGroup>

## Sandbox vs. production keys

| Property          | Sandbox                             | Production                      |
| ----------------- | ----------------------------------- | ------------------------------- |
| Base URL          | `https://api.sandbox.payglocal.in`  | `https://api.payglocal.in`      |
| Real money moved  | No                                  | Yes                             |
| Key prefix        | Typically begins with `sk_test_`    | Typically begins with `sk_live_` |
| Webhook events    | Simulated                           | Live                            |

<Tip>
Build and thoroughly test your entire integration in sandbox before switching to production keys. Sandbox transactions generate the same request and response shapes as production, making the switch seamless.
</Tip>

## Authentication error responses

If your API key is missing, malformed, or does not match the environment, the API returns one of the following errors.

### 401 Unauthorized

Returned when the `Authorization` header is absent or the token is not a valid API key.

```json
{
  "status": "FAILURE",
  "errorCode": "UNAUTHORIZED",
  "message": "Missing or invalid API key."
}
```

### 403 Forbidden

Returned when the API key is valid but does not have permission to perform the requested action (for example, using a read-only key to initiate a payment).

```json
{
  "status": "FAILURE",
  "errorCode": "FORBIDDEN",
  "message": "Your API key does not have permission to perform this action."
}
```
